Security Standards

Security Standards & Patterns

Security boundaries, rate limits, protections against abuse, and serverless architectural guards.

1. Bot Protection Triad

All user-facing forms (Contact Form, Feedback Form, Portfolio badge submission) implement three combined layers of bot defense, managed server-side via the checkBotProtection() helper in utils/security.ts:

Layer A: Cloudflare Turnstile

Verified against https://challenges.cloudflare.com/turnstile/v0/siteverify. If Turnstile tokens are invalid or missing, request fails closed with a 400 Bad Request.

Layer B: Honeypot Fields

A hidden form input field _hp is placed in templates. If automated bots fill out this field, the server returns a successful 200 OK response, but silently discards the submission without running any backend handlers.

Layer C: Submission Timing limits

Submissions occurring in under 3 seconds (as calculated by request parameters _t and _ts) are flagged as automated bot scripts. The endpoint returns a simulated successful 200 OK, silently dropping the payload.

2. API Rate Limiting

All Next.js serverless routes verify client request frequencies. Rates are tracked in-memory using a sliding-window algorithm defined in utils/rateLimit.ts, automatically pruning old logs every minute. When client IP addresses exceed their budget, the route returns an HTTP 429 Too Many Requests with a Retry-After header.

3. Gemini Input Size Limits

Before executing prompts inside the Google Gemini AI models, API routes clamp character inputs to prevent billing and token exploitation:

  • cvText (Resume text payload): Max 15,000 chars
  • jobDescription (Job specification content): Max 15,000 chars
  • submission (Portfolio project content): Max 20,000 chars
  • currentSkills: Max 2,000 chars
  • goal / query: Max 1,000 chars
  • targetRole / jobTitle: Max 200 chars
  • experienceLevel: Max 50 chars

4. HTTP Security Headers

Implemented globally via next.config.ts configuration:

  • X-Frame-Options: SAMEORIGIN — Clickjacking browser prevention.
  • Strict-Transport-Security — Forces HTTPS traffic for a duration of 1 year.
  • X-Content-Type-Options: nosniff — Stops MIME-type sniffing vulnerabilities.
  • Referrer-Policy: strict-origin-when-cross-origin — Sanitizes Referer info on cross-origin requests.
  • Permissions-Policy — Restricts unused browser features (geolocation, mic, camera).

5. HTML & XSS Protection

A. Resend Mail Sanitization

All variables injected into email bodies (e.g. contact form text, author names) are escaped via escapeHtml() in utils/security.ts before templates render, neutralizing HTML/CSS injections.

B. DOMPurify for WordPress Posts

Dynamic content fetched from WordPress is sanitized using a strict allowlist. Attributes such as inline styling, editor identifiers, and script sections are removed completely before being injected into page templates.

C. JSON-LD Escape Guards

To avoid attacks where users insert string values containing </script> inside metadata scripts, all JSON-LD output uses a replace query:

dangerouslySetInnerHTML={{ __html: JSON.stringify(schema).replace(/</g, '\\u003c') }}

6. Error Sanitization

All catch blocks are standardized across endpoints. Stack traces and SDK exception variables are strictly kept in the server environment (via console.error() for log review) and are never exposed to browser interfaces. Clients receive a generic message:

{ "error": "An unexpected error occurred. Please try again later." }

7. Supabase Client Guard

To prevent developers from accidentally importing the database client running the admin role key (SUPABASE_SERVICE_ROLE_KEY) into client-side files, a check is verified on initialization inside utils/supabase.ts:

if (typeof window !== 'undefined') {
  throw new Error('getSupabase must only be called server-side.');
}
