System Blueprint

System Architecture

Technical overview, file structures, and data flow patterns in Graduates Hub.

1. Tech Stack Overview

Graduates Hub is built on a highly optimized, modern Next.js stack, leveraging serverless architecture for fast deployment and scalability:

Next.js 16.2.4 & React 19
Framework with App Router structure, leveraging React Server Components (RSC) and server actions.
Supabase
PostgreSQL database, Row-Level Security (RLS), and API clients handling data persistence.
Google Gemini v1 SDK
AI integration via `@google/genai` to analyze user inputs, CVs, and code files.
Tailwind CSS 4
Modern visual styling, leveraging native features and PostCSS for speed and efficiency.

2. Directory Layout

The codebase follows a standardized Next.js structure:

graduateshub-hub-next/
├── app/                          # Next.js App Router (Pages & API endpoints)
│   ├── api/                      # Serverless API routes (Gemini, Supabase, Resend)
│   ├── blog/                     # Blog dynamic routes & listing page
│   ├── career-roadmaps/          # Career roadmaps static/dynamic pages
│   ├── interview-prep/           # Interview preparation guides
│   ├── portfolio/                # Portfolio builder and category listings
│   ├── proof/[id]/               # Public portfolio submission showcases
│   └── tools/                    # Interactive career tools (Gemini-powered)
├── components/                   # Reusable React components
│   ├── home/                     # Home page elements
│   ├── course/                   # Course display templates
│   ├── layout/                   # Global layout (Navbar, Footer, CookieConsent)
│   └── shared/                   # Common elements (Newsletter, Promos)
├── data/                         # Hardcoded site datasets (Fast static page builds)
├── lib/                          # Site-wide constants and SEO configuration
├── supabase/                     # SQL database schema and initialization scripts
├── types/                        # Main TypeScript interfaces (types/index.ts)
└── utils/                        # Utilities (rate limiter, safety guards, email escaping)

3. Key Architectural Patterns

A. Data-Driven Static Content (Zero Database Overhead)

Rather than pulling career data or course listings from a database dynamically on every request, all primary site content is stored inside /data as TypeScript datasets. This ensures high speed, low database billing, and robust static generation.

import { roadmaps } from '@/data/roadmaps';

export async function generateStaticParams() {
  return roadmaps.map((r) => ({ slug: r.id }));
}

export default async function Page({ params }: { params: Promise<{ slug: string }> }) {
  const { slug } = await params;
  const roadmap = roadmaps.find((r) => r.id === slug);
  if (!roadmap) return notFound();
  // ... render roadmap UI
}

B. Routing & Metadata Generation

Dynamic pages utilize the [slug] format. Every route defines a generateMetadata() export to load canonical URLs, OpenGraph (OG) cards, Twitter cards, and descriptions for SEO.

C. Component Separation

  • Layout components (components/layout/): Global layouts (e.g. Navbar, Footer, CookieConsent).
  • Feature components (components/home/, components/course/): Domain-specific files.
  • Shared components (components/shared/): Multi-page blocks like banners.

4. Database Schema

Supabase acts as our primary database, structured strictly for tracking portfolio projects and evaluations.

Row-Level Security (RLS)

The portfolio_proofs table enforces strict read-write access rules via policies in /supabase/schema.sql. Read operations are public, but write access is restricted only to backend routes running the private service role key.

Table Details: portfolio_proofs

  • id (text, Primary Key): Unique hash identifying the badge proof.
  • task_id & task_title & task_field (text): Information representing the completed project task.
  • graduate_name (text): Display name of the graduate who submitted the task.
  • submission (text): User's description or text proving task completion.
  • submission_links (jsonb): Array of verified links (e.g. GitHub repos, live sites).
  • evaluation (jsonb): JSON payload from the Gemini API containing grading score, code review summary, and comments.
  • created_at (timestamptz): Timestamp of the submission.

5. Third-Party Integrations

  • Google Gemini API: Orchestrated server-side via the `@google/genai` SDK to evaluate student submissions, build learning paths, and audit CVs.
  • Resend: Handles mail sending for feedback and contact pages.
  • OneSignal: Delivers browser-push notifications initialized through a lazy client script.
  • Cloudflare Turnstile: Secures form inputs by executing verification tokens before executing endpoints.
